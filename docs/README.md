# VoStory

`VoStory` 是一个面向团队协作的 AI 有声小说 / 广播剧制作平台。

它的目标不是单纯“把文字转成语音”，而是把小说内容结构化成可编辑、可复用、可批量生产的广播剧工程，支持角色资产管理、批量语音生成、局部返工、团队协作与最终导出。

## 项目命名建议

推荐仓库名：`vostory`

推荐理由：
- 简短，适合 GitHub 仓库名
- 含义明确，`Voice + Story`
- 既适合“有声小说”，也适合“广播剧制作”
- 后续如果做平台品牌，也比较自然

可选备选名：
- `storyvoice`
- `castnovel`
- `echo-novel`
- `dramacast-ai`
- `voiceforge-story`

如果你想偏中文语境或更偏“广播剧制作平台”，后面还可以再换展示名，但仓库名先用 `vostory` 会比较稳。

## 产品定义

VoStory 是一个服务化部署的广播剧制作平台，面向个人作者、制作团队、工作室和平台方，提供从小说导入、文本解析、角色管理、语音生成、人工修正到章节导出的完整生产流程。

一句话定义：

> 把小说内容转化为可编辑、可复用、可协作生产的广播剧项目。

## 目标用户

- 个人作者：想快速把作品变成多人有声内容
- 制作团队：需要多人分工处理脚本、角色、试听和返工
- 工作室：需要统一管理角色资产、项目进度和导出交付
- 平台方：需要规模化生产和复用音频资产

## 核心问题

产品要解决的不只是 TTS，而是整条制作链路里的这些问题：

- 自动抽取人物，并处理别名、称呼、重复角色
- 自动识别对白、旁白、内心独白和场景切换
- 保证同一角色在全书、跨章节、跨项目中的声音一致性
- 支持批量生成和局部重生成，避免整章返工
- 让团队成员可以分工修正脚本、试听审核和导出交付
- 让角色资产、发音词典、音频片段可以复用和继承

## 为什么不先做桌面端

如果产品从一开始就考虑团队协作、项目共享和资产复用，那么纯桌面端会在这些场景下变得吃力：

- 多人协同改稿
- 共享角色资产和发音词典
- 统一调度模型和 GPU 资源
- 批量任务管理和重试
- 项目审核、返工、权限控制
- 多本书、多团队、多项目管理

因此更适合的产品形态是：

`Web 前端 + 服务端 + AI/音频服务`

## 总体架构

推荐技术栈：

- 前端：`Vue 3 + TypeScript`
- 服务端：`Golang`
- AI/模型适配层：`Python` 或独立推理服务
- 数据库：`PostgreSQL`
- 缓存 / 队列：`Redis`
- 文件 / 音频存储：`MinIO` 或兼容 S3 的对象存储

推荐分层：

### 1. Web 前端

负责：

- 项目管理
- 章节与脚本编辑
- 角色管理
- 试听和重生成
- 任务状态查看
- 审核与导出

### 2. 业务服务层

由 Golang 提供，负责：

- 用户、组织、权限
- 项目与章节管理
- 角色资产管理
- 任务调度和状态管理
- 文件元数据管理
- 审计日志和版本记录

### 3. AI / 音频服务层

负责：

- 文本解析
- 人物抽取
- 台词归属
- 情绪标签生成
- TTS 调用
- 音频后处理
- 局部重生成

### 4. 存储层

负责：

- 业务数据
- 项目结构数据
- 角色资产
- 音频缓存
- 导出文件

## 推荐的产品核心对象

建议从第一版开始就把核心数据结构抽清楚：

- `Workspace`：组织或团队空间
- `Project`：一本书或一个制作项目
- `Chapter`：章节
- `Scene`：场景
- `ScriptSegment`：脚本片段，可能是旁白、对白、独白
- `Character`：项目内角色
- `VoiceProfile`：声音配置
- `VoiceAsset`：全局可复用声音资产
- `PronunciationDictionary`：发音词典
- `GenerationTask`：生成任务
- `AudioClip`：生成后的音频片段
- `ExportJob`：章节导出任务

## MVP 目标

MVP 不追求“全自动完成广播剧”，而是先打通一条最小可用的制作链路：

`导入小说 -> 自动切分脚本 -> 抽取角色 -> 绑定声音 -> 批量生成 -> 人工修正 -> 导出章节`

这条链路跑通，产品就已经具备明确价值。

## MVP 范围

第一版建议只做这些功能：

### 1. 项目管理

- 创建项目
- 导入 txt / docx / epub
- 按章节管理内容
- 查看项目处理状态

### 2. 文本解析

- 自动分章节
- 自动切分段落
- 初步识别旁白 / 对白 / 内心独白
- 初步抽取角色候选

### 3. 角色管理

- 展示角色列表
- 合并重复角色
- 编辑角色名称和别名
- 为角色绑定 voice profile
- 支持主角色、次角色、路人角色分层

### 4. 脚本编辑

- 按片段查看内容
- 修正说话人
- 调整片段类型
- 调整情绪标签
- 支持发音词典修正

### 5. 批量生成

- 按章节批量生成音频
- 单片段试听
- 单片段重生成
- 生成失败重试

### 6. 导出

- 章节音频拼接
- 基础停顿插入
- 音量标准化
- 导出 wav / mp3

## MVP 暂不做

这些能力可以明确放到后续版本，避免第一版过重：

- 自动 BGM / 音效编排
- DAW 级时间轴编辑
- 实时多人协同编辑
- 复杂审批流
- 多租户 SaaS 商业化能力
- 高级情绪曲线控制
- 自动导演式广播剧编排

## MVP 最终效果

一个合格的 MVP 完成后，用户应该能做到：

1. 上传一本小说
2. 自动得到按章节拆好的脚本
3. 自动得到角色候选和对白归属初稿
4. 手工修正角色、台词、情绪和发音
5. 给每个角色绑定一个固定声音
6. 批量生成章节语音
7. 对有问题的句子做局部返工
8. 导出可试听、可交付的章节音频

这时产品已经具备生产工具价值，即使距离“完整广播剧平台”还有距离。

## 最大技术难点

这个项目最难的地方，不是单点语音生成，而是下面这些系统性问题：

### 1. 角色抽取与去重

- 同一角色可能有多个名字、称呼、代词
- 误拆角色会直接导致声音错配

### 2. 台词归属

- 小说里很多对白没有直接写明说话人
- 多人连续对话很容易分错

### 3. 声音一致性

- 同一角色在长篇文本中必须稳定
- 角色跨章节、跨项目时要能复用

### 4. 长篇任务调度

- 一部小说可能很长
- 分析、生成、导出都需要断点续跑和缓存

### 5. 局部返工能力

- 修改一句台词，不应该重新跑整章
- 每个音频片段都需要可追踪来源和版本

### 6. 人工修正效率

- 模型永远会犯错
- 产品价值很大程度上取决于“修错有多快”

## 本地模型与在线模型策略

推荐采用混合架构，而不是只押注一种方式。

### 本地模型适合

- 人物抽取
- 对白 / 旁白分类
- 别名归并
- 情绪初标
- 多轮局部修正

优点：

- 成本可控
- 响应快
- 数据不必频繁上传

### 在线模型适合

- 更复杂的语义判断
- 更高质量的情绪理解
- 高质量商业 TTS
- 最终精修音频

优点：

- 效果通常更强
- 可快速接入成熟能力

### 推荐策略

- 文本解析优先本地
- 草稿音频优先本地
- 关键片段和最终成片可选云端精修

## 关于 Qwen 的建议

Qwen 系列可以重点考虑用于中文文本理解和结构化抽取，尤其适合：

- 人物识别
- 别名归并
- 对话归属初判
- 场景切分
- 情绪标签草稿生成

但要避免把整个产品效果押在单一模型上。更稳妥的做法是：

- 模型输出结构化中间结果
- 提供人工修正界面
- 做缓存和版本管理
- 保留替换模型的能力

## 开发注意点

### 1. 先设计中间结构，不要直接处理原文

系统不应该反复操作整本小说原文，而应该尽快把内容转成结构化对象：

- 章节
- 场景
- 脚本片段
- 角色
- 声音配置
- 音频片段

这会直接影响后面的可编辑性、缓存能力和重生成能力。

### 2. 模型调用一定要做适配层

不要把业务逻辑和某个模型 SDK 绑定死。

建议定义统一接口，例如：

```text
AnalyzeText(input) -> structured result
GenerateSpeech(segment, voice_profile, emotion) -> audio clip
PostProcessAudio(clips) -> final audio
```

这样未来替换本地模型、云端模型或 TTS 服务时不会推倒重来。

### 3. 从一开始就考虑版本追踪

这些对象都需要版本或变更记录：

- 脚本片段
- 角色绑定
- 发音词典
- 音频片段
- 导出结果

否则后面一旦返工，你会很难知道问题来自哪里。

### 4. 任务系统必须独立设计

生成任务一定不要直接挂在请求里同步执行。

需要：

- 队列
- 重试
- 超时
- 中断
- 恢复
- 状态追踪

因为长篇小说生产一定是异步任务。

### 5. 先做“高频修正点”而不是“自动化幻觉”

第一版里最值钱的不是炫技，而是把这些动作做顺：

- 改说话人
- 改角色映射
- 改发音
- 重生成一句
- 重新导出章节

### 6. 不要把角色资产只做成项目内数据

如果你后面想支持多本书复用，角色与声音资产需要分层：

- 项目内角色
- 全局声音资产
- 可复用的 voice profile 模板

### 7. 提前考虑版权与合规

需要明确这些边界：

- 小说版权授权
- 声音克隆授权
- 商用归属
- 敏感内容审核
- 私有化部署的数据安全

## 成功标准

MVP 是否成功，可以先看这几个指标：

- 一本 10 万字左右的小说是否能顺利导入并完成结构化处理
- 用户是否能在页面上快速修正角色和台词归属
- 单章节是否能稳定批量生成音频
- 修改少量内容后是否支持局部返工
- 最终是否能导出可试听的章节成品

## 当前结论

当前最合理的产品方向不是桌面单机工具，而是：

> 一个以服务部署为核心、以团队协作为目标、以角色资产和生产流程为中枢的 AI 广播剧制作平台。

在这个方向下，`Vue 3 + Golang` 是可行的主架构，前提是从第一天开始把模型能力做成可替换的独立适配层。
