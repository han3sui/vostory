# VoStory MVP 方案

## MVP 目标

MVP 不追求"全自动完成广播剧制作"，而是先打通一条稳定可用的生产链路：

```
导入小说 → 结构化脚本 → 抽取角色 → 绑定声音 → 批量生成 → 人工修正 → 导出章节音频
```

只要这条链路能够稳定运行，产品就已经具备实际价值。

## MVP 范围

### 1. 项目管理

- 创建项目
- 导入 `txt` / `docx` / `epub`
- 按章节管理内容
- 查看项目处理状态

### 2. 文本解析

- 自动分章节
- 自动切分段落为脚本片段
- 初步识别旁白、对白、内心独白
- 自动抽取角色候选
- 长文分批处理：按可配置字数上限切块，每块独立调用 LLM，支持单批次重试和合并

### 3. 角色管理

- 展示角色候选列表
- 合并重复角色
- 编辑角色名称和别名
- 为角色绑定声音配置
- 区分主角、配角、路人角色

### 4. 脚本编辑

- 按片段浏览文本
- 修正说话人归属
- 修改片段类型
- 调整情绪标签和情绪强度
- 维护发音词典
- 精准填充：LLM 拆分结果自动对齐回原文，确保不丢字不加字

### 5. 批量生成

- 按章节批量生成音频
- 情绪 + 强度双维度传参
- 单片段试听
- 单片段重生成
- 失败任务重试
- SSE 实时推送生成进度

### 6. 导出

- 拼接章节音频
- 插入基础停顿
- 音量标准化
- 导出 `wav` / `mp3`

## MVP 暂不做

- 自动 BGM / 音效编排
- DAW 级时间轴编辑
- 实时多人协同编辑
- 复杂审批流
- 多租户 SaaS 商业化能力
- 导演式自动广播剧编排
- 高级情绪曲线控制

## MVP 完成后的效果

一个可用的 MVP 完成后，用户应该可以：

1. 上传一本小说
2. 获得按章节拆分的脚本初稿
3. 获得角色候选和对白归属初稿
4. 手动修正人物、台词、情绪和发音
5. 为每个角色绑定固定声音（支持多情绪参考音）
6. 批量生成章节语音，实时查看进度
7. 只重生成有问题的句子
8. 导出可试听、可交付的章节音频

## 成功标准

- 一本约 10 万字的中文小说可以顺利导入并完成结构化处理
- 长文分批处理稳定，单批次失败不影响整体
- 用户可以高效修正角色和台词归属
- 章节级批量生成稳定可用
- 小范围修改后支持局部返工
- 能导出可试听、可交付的章节成品

## 开发阶段

### 第 1 阶段：主链路打通

**核心业务**：

- 项目创建与管理
- 小说导入（txt / docx / epub 解析）
- 章节自动切分
- 脚本片段存储（含旁白/对白/独白类型）
- 角色抽取结果存储（含别名、层级）

**基础设施（同步完成）**：

- 全部核心领域对象的数据库表设计
- AI 服务配置模块：LLMProvider / TTSProvider 的 CRUD、连通性测试、运行时切换
- Prompt 模板管理：内置默认模板 + 用户自定义，按类型分（character_extract / dialogue_parse / emotion_tag 等）
- LLM 调用日志：记录 Prompt 版本、模型、耗时、输入输出摘要

### 第 2 阶段：可编辑

- 脚本编辑页（片段级编辑、修正说话人、调整类型/情绪/强度）
- 角色编辑页（绑定声音配置、编辑别名）
- 发音词典（项目级 + 全局级）
- 精准填充能力（LLM 输出对齐回原文）
- 手工修正能力

### 第 3 阶段：可生成

- TTS 适配层（统一接口，支持多厂商：本地 TTS / 在线商业 TTS / 自定义兼容接口）
- 情绪 + 强度双维度传参（文本标签或向量均可）
- 多情绪参考音频（同一角色不同情绪使用不同参考音）
- 任务队列（基于 Redis，异步生成、重试、超时、状态追踪）
- SSE 实时推送任务进度
- 音频缓存与版本管理
- 单句试听与重生成
- 音色试生成（预览声音效果）

### 第 4 阶段：可交付

- 章节音频拼接导出
- 基础音频后处理（段落间停顿、角色切换停顿、音量标准化）
- 多轨音频合并
- 导出记录管理
- 失败重试

### 第 5 阶段：可协作

- 团队权限分工
- 审核流程
- 全局声音资产库（跨项目复用）

## MVP 风险点

- 多人对话场景下的人物抽取和台词归属容易出错
- 语音质量可能在不同角色之间波动较大
- 长任务需要稳定的分批处理、重试和断点续跑能力
- 如果人工修正体验不够好，产品价值会明显下降
- LLM 分批处理时跨批次的角色去重和对话连续性需要额外处理
